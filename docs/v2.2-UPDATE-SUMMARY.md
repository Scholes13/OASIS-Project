### Bug #5: Dashboard Sync Issue (RESOLVED ✅)

**Problem Reported by User**:
1. Dashboard tidak sinkron setelah switch BU (data masih pakai BU lama)
2. Dashboard blank/kosong saat data loading

**Root Causes**:
1. **Race Condition**: Livewire property belum hydrated saat event handler load data
2. **No Loading States**: Tidak ada visual feedback saat loading

**Solutions Implemented**:

#### Solution #1: Session as Single Source of Truth
```php
// PATTERN BARU - CRITICAL untuk semua event handlers!

// ✅ ALWAYS: Update session FIRST
public function handleBusinessUnitSwitch($businessUnitId): void
{
    // 1. Session first (persistent storage, always fresh)
    session(['current_business_unit_id' => $businessUnitId]);
    
    // 2. Property second (UI binding only)
    $this->activeBusinessUnitId = $businessUnitId;
    
    // 3. Clear cache
    $this->clearDashboardCache();
    
    // 4. Load data (reads from session, guaranteed fresh)
    $this->loadDashboardData();
}

// ✅ ALWAYS: Read from session in data methods
protected function getFilteredBusinessUnitIds(): array
{
    // Session first! Property as fallback
    $id = session('current_business_unit_id') ?? $this->activeBusinessUnitId;
    
    $businessUnit = BusinessUnit::with('children')->find($id);
    // ...
}
```

**Why This Works**:
- Session = Persistent (tidak tergantung Livewire lifecycle)
- Property = UI binding (may lag during hydration)
- Event handlers = Race condition prone

#### Solution #2: Multi-Tier Loading States

     wire:target="switchBusinessUnit,handleBusinessUnitSwitch"

## 🎯 New Best Practices (v2.2)

### 1. Session as Single Source of Truth Pattern ⭐ CRITICAL

```php
// ⚠️ USE THIS PATTERN IN ALL LIVEWIRE EVENT HANDLERS

public function handleEvent($newValue): void
{
    // 1. Update session FIRST
    session(['key' => $newValue]);
    
    // 2. Then update property (for UI only)
    $this->property = $newValue;
    
    // 3. Then load data (reads from session)
    $this->loadData();
}

protected function getData(): array
{
    // ALWAYS read from session first
    $value = session('key') ?? $this->property;
    return Model::where('field', $value)->get();
}
```

**Why**:
- Session = Persistent, always fresh
- Property = UI binding, may lag
- Event handlers = Race condition prone

**When to Use**:
- ✅ All Livewire event handlers
- ✅ All data loading methods called from events
- ✅ Business unit switching
- ✅ Filter updates
- ✅ Any state change that triggers reload

### 2. Multi-Tier Loading States Pattern

**Tier 1: Full-Screen Overlay** (blocks interaction)
```blade
<div wire:loading.flex wire:target="action" 
     class="fixed inset-0 bg-gray-900 bg-opacity-60 z-50">
    <div class="bg-white p-8">
        <x-loading-spinner size="lg" />
        <div>Loading...</div>
    </div>
</div>
```

**Tier 2: Skeleton Loading** (shows layout)
```blade
<div wire:loading class="animate-pulse">
    <div class="h-10 bg-gray-200 rounded w-16"></div>
</div>
<div wire:loading.remove>
    <p>{{ $data }}</p>
</div>
```

**Tier 3: Inline Spinners** (for buttons)
```blade
<button wire:click="action" wire:loading.attr="disabled">
    <span wire:loading.remove>Save</span>
    <span wire:loading>
        <x-loading-spinner size="sm" /> Saving...
    </span>
</button>
```

**When to Use**:
- Tier 1: Major state changes (BU switch, page navigation)
- Tier 2: Content loading (lists, grids, stats)
- Tier 3: Button actions (save, delete, submit)

### 3. Reusable Component Pattern

**For Traits**:
```php
// Create trait in: app/Livewire/Traits/
trait HasYourFeature {
    // Properties
    public $property = value;
    
    // Methods
    public function method() { }
    
    // Include PHPDoc with @example
}

// Use in component:
use HasYourFeature;
```

**For Blade Components**:
```blade
<!-- Create in: resources/views/components/your-component.blade.php -->
@props(['prop1' => 'default', 'prop2'])

<div {{ $attributes->merge(['class' => 'base-class']) }}>
    <!-- Component content -->
</div>

<!-- Usage: -->
<x-your-component :prop1="$value" prop2="value" class="extra-class" />
```

**When to Create Reusable Components**:
- ✅ Pattern used 3+ times
- ✅ Logic can be generalized
- ✅ Clear interface (props/methods)
- ✅ Self-contained functionality
